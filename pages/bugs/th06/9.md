## [what]
[hr_minor]

The bullet displacement bug is a bug where a boss's bullets are fired from a wrong position relative to the boss. This bug is encountered on Flandre and Patchouli.

## [how]
[hr_minor]

This bug happens occasionally on Patchouli's and Flandre's non-spells. More notably, this bug occurs around the same timestamps of the non-spells. 

## [why]
[hr_minor]

In order to understand how this bug takes place, we must understand how the game processes its events. Every frame, the game updates events in a specific order:

1. Moving boss according to its current velocity
2. Clamping boss position to its movement limits
3. Process scripted boss behavior
   a. Apply interpolation-based movement updates
   b. Handle bullet firing
4. Clamping boss position to its movement limits

### Boss velocity
[hr_minor]

The boss velocity is a 2D vector that represents by how much a boss moves in one game frame. For example, a vector of (1.0, -2.0) means the boss moves right by 1.0 units and down by -2.0 units (i.e. upward by 2.0 units).

### Boss movement limit
[hr_minor]

To prevent a boss from going off-screen, the boss is given a movement limit. This limit is a rectangle that represents the space the boss can exist in. The limit's (x,y) position as well as their (width, height) are usually set at the start of a new pattern. 

### Boss movement interpolation
[hr_minor]

Interpolation movement smoothly moves the boss from a starting position to a target position over a fixed duration. Different interpolation modes can be used to create movement patterns such as linear movement or smooth deceleration.

Each frame, the interpolation logic computes the boss's expected position along the movement path and derives a velocity from that position. However, on the final frame of the interpolation, the game [hl2]directly sets the boss position to the interpolation target position[/hl2] instead of continuing to derive it from velocity. This is likely done to ensure that the boss ends up in the correct final position.

### Bullet firing logic
[hr_minor]

Boss bullet firing is controlled by a timer-based system. Each frame, the firing timer is incremented, and once the timer reaches the configured firing interval, the boss fires its pattern and the shooting timer is reset.

When spawning bullets, the game uses the boss's current position combined with a given offset. As a result, bullets always spawn relative to the boss position when the firing logic executes.

### Where the bug occurs
[hr_minor]

The movement limit clamps (steps 2 and 4) prevent the boss from never visibly leaving its allowed movement area. However, a problem occurs because of interpolation-based movement updates happen between these two clamp operations.

On the final frame of an interpolation movement, the game directly sets the boss position to the interpolation target position. If this target position lies outside the movement limits, the boss temporarily exists outside its allowed area (step 3a).

Immediately afterwards (step 3b), the bullet firing logic executes. Since bullets spawn using the boss's current position, any bullets fired during this frame will use the boss's temporary out-of-bounds position as their spawn point. 

Of course, for this to happen, the shooting timer (step 3b) has to coincide with the final frame of the interpolation (step 3a).

After the bullets have been spawned, the game performs the second movement limit clamp (step 4). This moves the boss back within bounds before rendering occurs. As such, [hl2]the boss itself never visibly appears outside the movement area[/hl2], but bullets can still appear displaced from the boss.
